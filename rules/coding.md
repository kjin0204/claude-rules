# Coding Rules (공통 원칙)

## 코드 작성 원칙

- 동작하는 코드 > 완벽한 코드: 먼저 동작하게 만들고, 그 다음 개선한다
- 함수/메서드는 하나의 일만 한다 (단일 책임 원칙)
- 함수는 가능한 짧게 유지한다 (20~30줄 초과 시 분리 검토)
- 중복 코드는 3번 반복되면 추상화를 검토한다 (Rule of Three)
- 매직 넘버/문자열은 상수로 분리한다

## 네이밍

- 이름은 의도를 드러내야 한다 — 줄임말보다 명확한 이름 우선
- 변수/함수: `camelCase`, 상수: `UPPER_SNAKE_CASE`, 클래스: `PascalCase`
- boolean 변수/함수는 `is`, `has`, `can` 등의 접두사 사용 (e.g. `isVisible`, `hasError`)
- 함수 이름은 동사로 시작 (e.g. `getUser`, `updateProfile`, `validateInput`)

## 주석

- 코드로 의도를 표현할 수 있으면 주석 생략
- 주석은 **왜(Why)** 를 설명한다 — **무엇(What)** 은 코드가 설명한다
- TODO 주석은 `// TODO: <내용>` 형식으로 작성

## 에러 처리

- 에러는 묵살하지 않는다 — 반드시 로깅하거나 상위로 전파한다
- 사용자에게 노출되는 에러 메시지는 친절하게, 내부 에러 상세는 로그로만
- 예외는 예외적인 상황에만 사용한다 (흐름 제어 목적으로 사용 금지)

## 함수/모듈 설계

- 외부 의존성(API, DB 등)은 내부 로직과 분리한다
- 순수 함수(같은 입력 → 같은 출력, 부수효과 없음)를 우선 작성한다
- 전역 상태는 최소화한다

## 코드 리뷰 관점 (작성 시 자가 체크)

- 이 코드를 6개월 후의 내가 읽어도 이해할 수 있는가?
- 이 함수를 테스트하기 쉬운가?
- 불필요한 복잡성이 없는가?

## AI 협업 원칙

- 요청한 것만 변경한다 — 요청 범위를 벗어난 리팩터링/기능 추가 금지
- 기존 코드 스타일을 먼저 파악하고 일관성을 유지한다
- 변경 전 파일을 반드시 읽는다
- 불확실한 부분은 가정하지 않고 사용자에게 확인한다
